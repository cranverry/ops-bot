## 0. Executive Brief

**한 줄 정의:**
Spectrum OS는 Spectrum 조직의 인사·재무·생산·OKR·고객·확장 전략을 단일 인터페이스로 통합하여, COO가 30초 안에 모든 실행 판단을 내릴 수 있도록 만드는 비즈니스 운영 플랫폼이다.

**투자 근거:**
- 분기 RPE 목표 280M KRW 달성을 위해 현재 운영 병목(파편화된 데이터, 반복 소통, 수동 추적)을 제거한다
- 챗봇(OpsBot) + 대시보드(Mission Control) 이중 인터페이스로 조회·명령·자동화를 일원화한다
- 첫 배포 목표: **Phase 1 (OpsBot) 4주 내 운영 가동**

## 1. 전략적 맥락 (Strategic Context)

### 1.1 비즈니스 목표

| 레벨 | 목표 | 지표 |
|------|------|------|
| 연간 | 지속 가능한 크리에이티브 스튜디오 구조 확립 | 팀 규모 증가 없이 매출 2x |
| 분기 | RPE 280M KRW 달성 | 분기 누적 매출 ÷ 내부 인원 환산값 |
| 스프린트 | KR 달성률 80%+ 유지 | Sprint별 KR 추적 (현재 Sprint 5: KR1 49%, KR3 80%) |

### 1.2 현재 상태 (As-Is)

```
데이터 소스      도구               문제
─────────────────────────────────────────────────
파이프라인 현황  ClickUp (432건)   수동 쿼리 필요. 브랜드 폴더 아카이빙 전환 중
팀원/R&R        Notion R&R DB     8명 정의됨. 프리랜서 20명+ 별도 관리
OKR/KR          Notion CR팀 워크스 스프린트별 수동 업데이트. 실행과 단절
재무             미정              스프린트 매출 추적 없음. RPE 계산 불가
CS/VOC          미정              CS 처리 비정형. VOC→기획 피드백 루프 없음
지식(SOP)       Notion + 파편     7건 수집. 실시간 갱신 없음
```

### 1.3 문제 정의 (Problem Statement)

**핵심 문제:** 실행 판단에 필요한 데이터가 3개 이상의 도구에 흩어져 있어 COO의 의사결정 사이클이 느리다.

```
현재 플로우:
COO가 "OAS 현황이 어떻게 돼?" 질문 시
  → ClickUp 열기 (2분)
  → 필터 설정 (1분)
  → Notion 확인 (2분)
  → 종합 판단 (5분)
  = 10분+ 소요

목표 플로우:
  → OpsBot에 질문 (10초)
  = 30초 이내 답변
```

### 1.4 근본 원인 (Root Cause)

```
R1. 데이터 소스 파편화
    → ClickUp, Notion, Discord, 스프레드시트가 독립적으로 운영됨
    → 단일 쿼리 레이어 없음

R2. 운영 지식의 암묵지화
    → SOP가 개인 DM·구두 인수인계에 종속됨
    → 확장 불가능한 구조

R3. OKR-실행 단절
    → 스프린트 KR이 일간 태스크 실행과 연결되지 않음
    → 달성률이 결과 확인용에 그침

R4. 재무 가시성 부재
    → 매출·비용·RPE가 실시간으로 집계되지 않음
    → 리소스 배분 판단의 근거 데이터 없음
```

## 2. 제품 비전 및 원칙 (Product Vision & Principles)

### 2.1 비전

> "Spectrum의 모든 운영 데이터를 하나의 뇌로 연결하고, 자연어 한 문장으로 판단하고 실행한다."

### 2.2 설계 원칙

| 원칙 | 의미 | 위반 예시 |
|------|------|-----------|
| **Single Source of Truth** | 모든 데이터는 하나의 소스에서 흘러야 한다 | 동일 데이터를 2곳에 수동 입력하는 구조 |
| **Role-Gated Visibility** | 각 역할은 판단에 필요한 데이터만 본다 | L3 외주가 전체 매출을 볼 수 있는 구조 |
| **Action over Report** | 조회가 아닌 실행까지 한 인터페이스에서 완결 | "확인 후 ClickUp에서 업데이트" 요구 |
| **Zero Repetition** | 사람이 반복하는 일은 시스템이 처리한다 | 매 스프린트 동일 온보딩 설명 반복 |

### 2.3 범위 외 (Non-Goals)

- 외부 고객(구매자)이 직접 사용하는 인터페이스 — 아님
- Booth.pm 플랫폼 대체 — 아님
- 범용 프로젝트 관리 도구 — 아님. Spectrum 전용 운영 OS임

## 3. 사용자 세그먼트 (Users & Jobs-to-be-Done)

### 3.1 역할 정의

| 역할 | 실제 인물 | 핵심 Job-to-be-Done |
|------|-----------|---------------------|
| **L0 Owner** | 박범진 (COO) | "지금 당장 어디가 막혀있고, 다음 30분에 뭘 해야 하는가" |
| **L1 Admin** | 최현세, 임주영 등 리드 | "내 브랜드/유닛은 어떤 상태이고, 누가 뭘 해야 하는가" |
| **L2 Member** | 한동우, 남태경, 윤건 등 | "오늘 내가 해야 할 일이 뭐고, 기준이 뭔가" |
| **L3 External** | 프리랜서 20명+ | "내 브리프가 뭐고, 납품 기준이 뭔가" |

### 3.2 L0 Critical User Journey (COO 관점)

```
[아침 시작] → OpsBot: "오늘 현황 브리핑"
               ↓
   ┌───────────────────────────────────────┐
   │ 스프린트 현황: OAS 리깅 완료 대기 2건  │
   │ KR1: 6,622/7,000 (49%)               │
   │ 버퍼: OAS 2개 (임계값 근접)           │
   │ CS 미처리: 3건                        │
   │ ⚠️ 블로커: MRG 텍스처 지연 (임주영)   │
   └───────────────────────────────────────┘
               ↓
[의사결정] → "MRG 텍스처 홀드, OAS 기획 재개"
               ↓
[실행] → ClickUp 자동 업데이트 + Discord 알림
```

### 4.2 데이터 모델

```
핵심 엔티티

Task (ClickUp 기반)
├── id, name, status, priority
├── assignee → Member.name
├── list → Sprint (2주 주기)
├── folder → Unit (기획/생산/출시)
└── tags, due_date, description

Member (Notion R&R DB 기반)
├── name, role(multi_select)
├── responsibilities[]
└── level: L0~L3

Sprint (Notion CR팀 기반)
├── number, period
├── KR[] { id, title, target, current, unit }
└── tasks[] → Task[]

FinancialRecord (Google Sheets 기반)
├── sprint, brand
├── revenue_krw, revenue_jpy
└── rpe, headcount

Buffer (ClickUp 집계)
├── brand
├── count (QA완료~출시준비)
└── threshold: 3 (기획 홀드 트리거)
```

### 5.1 Phase 1 — OpsBot (챗봇) `목표: 4주`

#### P0: 운영 조회 (Operational Query)

**근거:** R1(파편화) 해결. COO 10분 → 30초 달성의 핵심

| 기능 ID | 기능 | 입력 예시 | 출력 | 데이터 소스 |
|---------|------|-----------|------|-------------|
| F-01 | 파이프라인 현황 조회 | "오늘 현황 브리핑" | 스테이지별 태스크 수, 블로커, 담당자 | ClickUp |
| F-02 | 버퍼 상태 조회 | "OAS 버퍼 얼마야?" | 브랜드별 비축량 + 임계값 대비 | ClickUp 집계 |

| F-04 | 팀원/역할 조회 | "리거 누구야?" | 이름, 역할, 현재 태스크 | Notion R&R DB |
| F-05 | SOP 조회 | "모델링 납품 기준 알려줘" | 관련 문서 청크 + 출처 | Notion KB |

#### P0: 파이프라인 명령 (Command Execution)

**근거:** R1 해결 + "Action over Report" 원칙

| 기능 ID | 기능 | 입력 예시 | 실행 | 권한 |
|---------|------|-----------|------|------|
| F-06 | 태스크 스테이지 이동 | "[태스크명] 리깅 완료 처리" | ClickUp status 업데이트 | L2+ |
| F-07 | 버퍼 기획 홀드/재개 | "BTQ 기획 홀드" | ClickUp 플래그 + Discord 알림 | L1+ |
| F-08 | 온보딩 링크 생성 | "모델러 온보딩 링크 만들어줘" | JWT 임시 링크 생성 + DM 발송 | L1+ |
